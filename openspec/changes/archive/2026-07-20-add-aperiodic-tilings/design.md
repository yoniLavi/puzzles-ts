# Design — add-aperiodic-tilings

## Context

Change 1 (`extend-grid-tilings`) landed the 14 periodic tilings behind a
deliberate RNG boundary. This change crosses it: the four aperiodic tilings are
RNG-bearing, desc-round-tripping, and carry ~3,500 lines of generated lookup
tables.

Four surveys of the C informed this design (hat, spectre, penrose, and the
aperiodic side of `grid.c`). Their findings are recorded as decisions below
rather than left to be rediscovered mid-port — the same discipline change 1
used, and the reason its addendum could be written honestly.

The volume here is deceptive. Only ~2,400 lines are *logic*; the rest is table
data that must not be typed by a human. And of that logic, the genuinely subtle
parts are small and identifiable: two mutual recursions, one BFS with a
half-tile pairing rule, and three places where the RNG stream must match
draw-for-draw.

## Decisions

### D1 — Split the differential in two: geometry-from-desc, then desc-from-seed

This is the central verification decision, and it is change 1's D1 applied one
level down.

An aperiodic grid is a function of `(type, width, height, seed)`, so a naive
differential has two failure causes braided together: wrong geometry and a
wrong RNG draw order. Untangle them by testing each alone.

- **Part A — geometry, RNG-free.** Feed TS a **C-recorded desc** and assert the
  full incidence dump (dots, edges, faces, indices) matches the C exactly. The
  existing `grid-trace.c` already accepts `<type> <width> <height> [desc]`, so
  this needs a fixture-matrix extension, not a new harness. This exercises all
  the recursion, the metamap/hexmap stepping, the half-tile pairing and
  `gridTrimVigorously` with **zero seed plumbing**.
- **Part B — RNG fidelity.** Given a seed, assert TS's `gridNewDesc` emits the
  C's desc **string** byte-for-byte. The desc is a short string, so this
  fixture is tiny and a failure names the draw order directly.

A failure in A is wrong geometry; a failure in B is a wrong draw. Without the
split, a red test means "somewhere in 2,400 lines of recursion, or one
misordered `randomUpto`."

Part A is also the stronger of the two in the same sense change 1's D7 was:
dot indices are assigned in first-encounter order driven by the generator's own
BFS, so index-exact agreement proves the **emission order** matches, not merely
the resulting shape.

### D2 — A random draw is an observable side effect; never short-circuit one

All three tilings pick weighted-random candidates by linear scan, and **all
three call `randomUpto` unconditionally even when the candidate list has exactly
one entry**:

| Site | Single-entry tables |
|---|---|
| `hat.c:421` `choose_mpp` | `parents_T` (1 entry) |
| `spectre.c:64` `choose_poss` | `poss_J`, `poss_L` (1 entry each) |
| `penrose.c:505` `penrose_choose_random` | — (but same shape) |

Skipping the draw when `n === 1` is the obvious optimisation and it is **wrong**:
it desynchronises the stream and yields a different — still entirely valid,
entirely plausible-looking — tiling. Nothing asserts, nothing throws.

The same rule extends to the weight constants. They are exact integer
approximations to algebraic numbers (`PROB_F = 17459667` ≈ a function of √5;
spectre's `PROB_*` ≈ functions of √15). **Transcribe the integers verbatim; never
recompute them from a square root.**

And transcribe upstream's oddities as-is. `hat.c:374`'s `starting_hats` entry
for `TT_T` uses `PROB_P`, not `PROB_T`. Whether that is intentional or an
upstream typo is irrelevant to us: it is what the C build draws against, so
"fixing" it diverges every hat desc.

This is the same family as change 1's addendum finding about
`greatgreatdodecagonal`'s asymmetric boundary guards — **upstream asymmetries
that read as sloppiness are load-bearing until proven otherwise**, and the port
is not the place to prove it.

### D3 — Tables are generated by script and checked in; never hand-typed

~3,500 lines across three sources, with three different extraction problems:

1. **`hat-tables.h`** (2,120 lines) — plain C array initialisers, no macros at
   all, rigidly regular formatting (every kitemap data line is exactly four
   `{a,b,c}` entries). A ~30-line regex script over the located table spans
   extracts it. Upstream itself generates this file (`hatgen --tables`), which
   is the strongest possible argument that no human should retype it.
2. **`spectre-tables-{auto,manual}.h`** (1,380 lines) — the *bodies* are plain
   literals, but the table **names** are X-macro expansions of
   `HEX_LETTERS(Z) = Z(G) Z(D) Z(J) Z(L) Z(X) Z(P) Z(S) Z(F) Z(Y)`, and entries
   reference `HEX_*` enum values and `PROB_*` macros. Preferred extraction is
   **compile-and-print**: a throwaway C program that includes the headers and
   walks `hexdata[]`, emitting JSON. That is immune to formatting drift and
   gives ground truth for the `bool internal` field. A textual scrape is the
   fallback; if used, checksum it against the compiled route once.
3. **Penrose has no table file.** Its "tables" are two nested `switch`
   statements — `transition` (60 leaves) and `transition_in` (36 leaves) — which
   are control flow, not data. These are **hand-transcribed** into typed object
   maps, because that is where TS actually earns its keep: keying them on a
   `type Letter = 'A'|'B'|'U'|'V'|'C'|'D'|'X'|'Y'` string-literal union makes
   the transcription **exhaustively type-checked**, and 96 leaves is small
   enough to review. The residual typo risk is covered by D1's part A.

Output shape for hat/spectre: **flat `Int8Array`s** with the same index
arithmetic upstream uses (`kitemapIndex(step, kite, hat, meta) = step + 4*(kite +
8*(hat + 4*meta))`). All values are in `[-1, 12]`. This avoids allocating 5,376
objects for the kitemaps alone.

Each generated `.ts` keeps its generator script checked in beside it, plus a
header comment pointing at both the script and upstream's own generator. The
file is not human-maintainable and must say so.

### D4 — The `"dummy"` replay RNG is a mandatory bit-exact port target

Both penrose (`penrose.c:583-591`) and spectre (`spectre.c:309-330`) lazily
create `random_new("dummy", 5)` when **replaying a stored desc** whose
coordinate string ran out. It exists because the obvious deterministic choice
risks unbounded recursion in the step function; a fixed-seed PRNG breaks the
symmetry.

Divergence here is **silent in the worst way**. The result is a perfectly valid,
self-consistent patch: `paramsInvalid` still passes, `makeConsistent` still
succeeds, nothing asserts. The observable symptom is that **the same desc yields
a different grid**, so a saved Loopy game loads with its clues attached to the
wrong faces rather than failing to load.

Requirements, all three of which are load-bearing:

- seed with `randomNew("dummy")` — the C's `5` is the byte length of the string,
  not a second seed component;
- create it **lazily at the same call site** (not eagerly at context creation),
  because an eager construction that never gets used still changes nothing, but
  an eager one that gets used *earlier* draws differently;
- **share it across all subsequent extensions** in that context.

It does not trigger on the normal path — `randomise` stores exactly the depth it
reached — but it does trigger when a desc is replayed at a larger `w`/`h` than it
was generated for. That case gets a **dedicated targeted test**, since no
ordinary fixture reaches it.

### D5 — Bound the unbounded loops, but only as a divergence tripwire

`hat.c:691` is `for (depth = 2;; depth++)` with no exit condition. It terminates
because each iteration invents a new enclosing metatile and at sufficient depth a
rewrite always exists. Real depth is O(log area) — tens, not thousands.

Apply this repo's existing `engine/retry-limit.ts` treatment: a generous cap that
a faithful port never reaches, throwing `RetryLimitExceeded` rather than
returning a fallback. Per that module's own doctrine, the cap **cannot move a
generated board** — exhaustion throws, so no seed that used to converge can
quietly start producing a different desc.

The alternative (leave it unbounded, matching C literally) is unacceptable here
for the reason `retry-limit.ts` documents at length: these generators are
synchronous, so a non-terminating loop owns its thread outright — it hangs a
browser tab in production and orphans a vitest worker that pins a core for ever
in test.

`penrosectx_step_recurse` and `spectrectx_step_hex` also recurse without a
structural bound, but they ascend a lazily-grown hierarchy rather than looping,
so JS stack depth bounds them naturally at these magnitudes. They get an assert,
not a cap.

### D6 — `gridTrimVigorously` becomes an adjacency index: O(n²) → O(edges)

Change 1 recorded this as D8, expecting a `Map` keyed on the ordered dot pair to
be a memory optimisation. The survey shows it is **required, and the win is
asymptotic, not just spatial**.

Upstream builds `dotpairs[a*n + b]` = the face in which dot `a` is immediately
followed by dot `b` in its clockwise ring — a directed-half-edge → face map. The
decisive finding: **the stored face index is never read.** `dotpairs` appears on
the right-hand side of nothing but a `>= 0` presence test. So the matrix is
morally a `Set` of directed half-edges wearing an `int[]` costume.

It answers exactly two queries:

1. **landlocked(`i`)** ⟺ for every `j`, `present(i,j) == present(j,i)` — i.e.
   `i`'s successor set equals its predecessor set, which is precisely "every edge
   at `i` has faces on both sides".
2. **merge** landlocked pairs `(i, j)` with `j < i` where both directions are
   present.

Both are set operations on a dot's *neighbours*, but upstream scans all `n²`
cells unconditionally for each. So build the incidence directly in the same pass
over faces:

```
succ: Map<dotIndex, Map<dotIndex, faceIndex>>
pred: Map<dotIndex, Set<dotIndex>>
```

Then landlocked(`i`) is `succ[i].size === pred[i].size && every b of succ[i] is in
pred[i]` — O(deg i) — and the merge loop walks `succ[i]` rather than all `j`.
Both drop from O(n²) to O(edges).

Semantics are identical, because presence is all the C ever tests.

**Why this is not optional.** Dots scale O(w·h), so the dense matrix scales
O((w·h)²). A 10×10 spectre patch is ~450–500 dots ≈ 1 MB — tolerable. A 25×25 is
~3,000 dots ≈ 36 MB, and a 50×50 is ~12,000 dots ≈ **576 MB**, which does not run
in a browser worker. Upstream gets away with it because its grids are small and
it is not running in a tab. The quadratic *time* cost bites at the same rate.

### D6a — Trimming renumbers, so `index` cannot stay `readonly`

`gridTrimVigorously` compacts faces and dots in place, preserving relative order
and **rewriting `index` to the new dense numbering**. But `GridDot.index` and
`GridFace.index` are `public readonly` in `grid-core.ts` — a change-1 decision
that was correct when nothing renumbered.

Drop `readonly` on `index` for both. The alternative — rebuilding fresh
`GridDot`/`GridFace` objects and remapping every face→dot reference — risks
breaking the dot **identity sharing** that `makeConsistent` depends on
(deduplicated corners must remain the same object), which is a much worse failure
than a mutable field.

Narrow the mutation to trimming: `index` is assigned at construction and by the
compaction pass, nowhere else. Say so in the field comment, since the `readonly`
was load-bearing documentation.

### D6b — An empty trimmed grid throws rather than returning silently

If nothing is landlocked, upstream's `j` stays `-1`, no face is selected, and the
grid is emptied. The **legacy** Penrose path explicitly checks for this and
returns `NULL` (`grid.c:3215-3226`); the four modern paths do **not**, and would
hand back an empty grid.

Throw explicitly. An empty grid downstream in Loopy is a puzzle with no faces to
clue — a confusing failure far from its cause. And keep the "no landlocked
component" sentinel as a nullable, not `-1`, so an empty case cannot accidentally
match `canonify(d) === -1`.

### D7 — Port `nTimesRootK` exactly; do not substitute `Math.round(n * Math.sqrt(k))`

`misc.c:569` computes `round(n·√k)` by an exact bitwise square-root algorithm,
written that way *specifically* to avoid FP rounding — upstream's comment names
the triple-rounding risk (rounding √k, rounding the product, rounding to
integer). It is the bridge where penrose's exact ℤ[√5] arithmetic and spectre's
ℤ[√3] arithmetic finally become integer pixels, once, at the last possible
moment.

For the magnitudes involved the FP substitute gives identical answers today. Port
the exact algorithm anyway: it is 32 iterations of trivial integer work, and the
substitute silently reintroduces exactly the risk the algorithm was written to
remove — while making the C reference fixtures less trustworthy as a check.

It is **shared** (penrose k=5, spectre k=3; hat does **not** use it — hat's grid
path is pure integer units), so it lands in a neutral engine module, not inside
a tiling-specific file.

C's sign handling (`INT_MIN + (int)(-x - (unsigned)INT_MIN)`) is an
overflow-UB-avoidance dance with no TS analogue; `sign * x` is correct. The odd
expression is intentional upstream, not a bug to preserve.

### D8 — Normalise negative zero at one choke point, not with scattered `|| 0`

Change 1's addendum names this the live hazard, and it is: it nearly shipped in
floret, produced a **structurally perfect** grid, and was invisible to every
check except a structural one.

All three aperiodic tilings use signed basis vectors and reach `-0` the same way
floret did — `pointscale(-1, {0,0})` in hat, `-y.cr3` at `spectre.c:536`,
sign-multiplication throughout penrose.

Rather than sprinkling `|| 0` through the arithmetic (which is easy to
under-apply and impossible to review), **normalise once at the tiling→grid
callback boundary** — the single place where exact-arithmetic coordinates become
grid pixel coordinates. That is one reviewable choke point per tiling, it is
where the value stops being an intermediate, and it is exactly where D1's part-A
differential observes.

Note the dedup path is *already* safe by accident (`` `${-0}` === "0" ``, and
`Map` keys use SameValueZero). That is precisely what makes this dangerous: the
grid comes out correct and only `Object.is` disagrees.

### D9 — Drop `penrose-legacy.c`; reject a `'G'` desc explicitly

Confirmed by survey: legacy is reached only via `if (*desc == 'G')`
(`grid.c:3370`, `:3412`), and `grid_new_desc_penrose` can never emit it —
`desc[0]` is `'0'+random_upto(rs,10)`, so always a digit. It is also the only
float-bearing tiling code, so dropping it makes the entire Penrose path
exact-integer.

Reject `'G'` with an **explicit** error naming legacy descs. Falling through
would reach the generic "expected digit at start of grid description" path,
which is survivable but misleading to whoever hits it.

Cost: pre-fork saved games carrying a legacy Penrose desc will not load. Per the
project's stated position that old saves and pre-pivot shared IDs are
expendable, that is accepted.

### D10 — `tree234` → `Map`/`Set`; BFS queue → array with a cursor

All three tilings use `tree234` purely as a **visited set** keyed on geometry
(penrose's `penrose_cmp`, spectre's `spectre_cmp` — both compare only the first
two vertices, which force the rest). Membership only: `add234`/`find234`, plus a
bulk drain for cleanup. **The tree's ordering is never observed**, so a `Map`
keyed on the vertex coefficients is exact.

The one trap: emission order comes from the **BFS queue**, not the tree. Do not
iterate the set to emit faces.

Penrose's queue is an intrusive linked list that is never dequeued — the cursor
just advances. That is an array plus an index (`for (let i = 0; i < queue.length;
i++)`, the array growing during iteration), **not** `shift()`.

### D11 — `coordSign` keeps JS doubles: a deliberate divergence from C bit-semantics

`coord_sign` decides the sign of `a + b√k` exactly, by comparing `c1²` against
`k·cr5²`. In C those squares can overflow 32-bit `int` for a large enough patch —
a real, if remote, upstream fragility.

JS doubles are exact to 2⁵³, comfortably beyond anything a grid produces. So the
correct port **does not** wrap these in `| 0` or `Math.imul`: doing so would
*introduce* the overflow that JS naturally avoids.

Called out explicitly because it inverts this change's usual rule. Everywhere
else, matching C's integer semantics is the priority (D2, D7, and `Math.trunc`
below). Here, deviating is more correct — and the difference is that this
arithmetic is a *predicate on exact values*, not a value that feeds a desc or an
RNG draw. Nothing downstream can observe the wider intermediate except by being
right more often.

### D12 — Drop spectre's 3-colouring sidecar

`spectre.c:381-403` maintains `hex_colour`/`prev_hex_colour`/`incoming_hex_edge`.
Survey confirms the only consumers are `auxiliary/spectre-help.c` and
`auxiliary/spectre-test.c`, both CLI-only; `grid.c` never reads them. The block
writes only those three fields — it does not touch `sc->c[]`, `sc->index`,
`*outedge`, or the RNG — so removing it is observationally neutral on the
generated tiling. Upstream's own comment says it is maintained unconditionally
"because it's easier than making it conditional."

### D13 — Order the port hat → spectre → penrose

Easiest control flow first, so the table-transcription approach (D3) is
established and proven before the hard recursion.

- **hat** — a 14-state serpentine kite enumerator; no BFS, no `tree234`, and
  its tables are the most mechanically extractable. Establishes D3.
- **spectre** — mutual recursion `step` ↔ `stepHex`, X-macro'd tables, the
  `"dummy"` fallback. Establishes D4.
- **penrose** — genuine recursion, BFS with a visited set, and the half-tile
  pairing. Logically hardest, and by then D1/D3/D4/D10 are all proven.

### D14 — The desc dispatch is new surface, and generation stays separate from construction

`gridNewDesc`/`gridValidateDesc` do **not exist in TS at all** yet — change 1
implemented the "reject a non-null desc" rule inside `gridNew`, but never the
dispatch itself. So this change builds both from scratch, including the trivial
triangular arm (`"0"`) and the "not used with this grid type" arm for the other
13. Triangular's validator currently lives inside `gridNewTriangular` and must be
pulled out to stand alone.

Preserve upstream's structural split, which is the useful part:

- **`gridNewDesc(type, w, h, rng)` is the only RNG-bearing function.**
- **`gridNew(type, w, h, desc)` is a pure deterministic function of its
  arguments.**

That separation is exactly what makes D1's two-part differential possible, so it
is a contract, not an implementation detail. The C makes the same promise and
asserts `gridValidateDesc(...) === null` on entry to `gridNew` — treating an
invalid desc reaching construction as a programming error. Keep that assert.

### D14a — `gridValidateParams` must stop throwing for aperiodic types

It currently throws for any non-periodic type. Remove that guard and extend the
existing `OBJECT_BOUND` table, which generalises cleanly: penrose
`{multiplier: 36}`, hats `{multiplier: 6}`, spectres `{multiplier: 49}`, all
`corners: false`.

Also reconcile the message. The TS port says `"Grid size must not be unreasonably
large"`; the C says `"Grid must not be unreasonably large"`. Decide deliberately
rather than by accident — if any fixture compares strings, they must agree; if
not, the divergence should still be recorded rather than left as a silent drift
from the reference.

### D15 — File layout

`grid.ts` stays the public barrel. The three tilings are large enough to warrant
their own directory rather than four more `grid-tilings-*.ts` siblings:

```
src/native/engine/tilings/hat.ts            hat-tables.ts       (generated)
src/native/engine/tilings/spectre.ts        spectre-tables.ts   (generated)
src/native/engine/tilings/penrose.ts        (maps inline — typed, reviewable)
src/native/engine/tilings/README.md         (points at the generator scripts)
scripts/gen-tilings-tables.<ext>            (checked in beside its output)
src/native/engine/n-times-root-k.ts         (shared: penrose k=5, spectre k=3)
```

`grid-tilings-aperiodic.ts` holds the four `gridNew*` wrappers, the desc
round-trip and `gridTrimVigorously` — the `grid.c` side, kept separate from the
three tiling engines it drives.

## Traps inventory

Small, specific things that cost a debugging session each if rediscovered. Not
decisions — just findings that must survive into the code as comments or asserts.

**Arithmetic**

- `hat.c:834`'s `(v.x*2 + v.y)/3` is C truncating division on freely-negative
  inputs. Use `Math.trunc`. The quotient is provably always exact (the invariant
  `2x + y ≡ 0 (mod 3)` is closed under every kite step), so assert it in dev
  rather than assuming it.
- `point_rot`'s `s % 10; if (s < 0) s += 10` — C and JS `%` agree here (both
  truncate). Comment it so nobody "fixes" it into a floor-mod.

**Desc parsing (untrusted input)**

- Both `grid_desc_to_penrose_params` and the spectre equivalent compute
  `strlen(desc) - 2` **before** checking the length, underflowing `size_t` on a
  1-char desc. Check length up front.
- `hat_tiling_params_invalid`'s `for (i = ncoords - 1; i > 1; i--)` on `size_t`
  is safe only because the `ncoords < 3` early return precedes it. Keep the
  ordering.
- `penrose_valid_parents` returns NULL for an unknown letter and the caller
  feeds it straight to `strchr` — safe only because `penrose_valid_letter` runs
  first in the same iteration. Make the TS lookup total instead of relying on
  order.

**Control flow**

- `spectrectx_step` uses `while (!m->internal)`, not `if`. `specin_S` is the
  only table with non-internal entries (exactly 4 of them). Collapsing it to an
  `if` is wrong **only on S-hexes** — a needle-in-haystack bug that D1's
  differential catches and eyeballing would not.
- Penrose's inner `switch (edge)` blocks have **no `default`**, so an
  out-of-range edge falls through into the next child's dispatch. Unreachable in
  practice; in TS, throw rather than mimic the fallthrough.
- `hat.c:54` reads `last_index`/`last_step` before they are ever written. Benign
  (states 5 and 11 are the only consumers and cannot be first), but initialise
  to 0 explicitly.
- `try_step_coords_kitemap` reads `unsigned kite/hat/meta` from `int` fields that
  legitimately hold `-1` elsewhere. Safe only because `extendCoords(4)` ran
  first. Add an explicit non-negative assert — this is a genuine latent-UB guard
  that the C leaves to invariant.
- The kitemap "impossible" sentinel is tested on `.kite` **only**, though `.hat`
  and `.meta` are `-1` in the same entries.
- `null_output_tile` is **not** dead code: `penrose_tiling_randomise` runs the
  entire BFS with a no-op callback purely to grow the prototype to the depth the
  desc must record.

**Geometry**

- **Hats' callback has a variable vertex count.** Unlike penrose (always 4) and
  spectre (always 14), the hat callback takes `nvertices` and the API does not
  promise a constant. Drive the face order from `nvertices`, not a literal.
- **Every spectre face is order 14 and includes a deliberately collinear
  vertex** — `spectre_angles[10]` is a straight 0° turn, the "double edge". It
  is how adjacent spectres share vertices correctly. Do **not** simplify
  collinear points out.
- **Recentring applies to penrose and spectres, but not hats.** Hats' bounding
  box is whatever survives trimming and will generally *not* equal
  `gridComputeSize`'s reported extent. That asymmetry is upstream's; do not
  "fix" it. Where recentring does apply, `(w - (highest - lowest))/2` can have a
  **negative** numerator when the surviving patch is wider than the promised
  rectangle — so `Math.trunc`, not `Math.floor`, or the whole grid shifts a
  pixel.
- **`nTimesRootK` is symmetric about zero, not floored**: sign is taken off the
  input, the magnitude is rounded, the sign reapplied. And it must return `+0`
  for `n === 0` — a TS implementation reaching for `Math.sign` gets `-0` for
  `-0`, feeding D8's hazard straight into a dot coordinate.
- Penrose's grid glue **transposes x and y** (`grid.c:3396-3399`) and `api_size`
  crosses the units to match (`grid.c:3288-3297`), then calls `generate(w=size.h,
  h=size.w)`. The two crossings cancel; any half-fix produces a wrong-aspect
  grid. Preserve verbatim.
- **The rational and irrational parts are scaled separately, then summed**:
  `unit*a + nTimesRootK(unit*b, k)`, *not* `unit * (a + nTimesRootK(b, k))`.
  Factoring the unit out changes where the single rounding happens and quietly
  defeats D7's whole point. Hats needs none of this — its coordinates are
  already integers on a triangular lattice.
- The half-tile pairing predicate is `edge == sibling_edge && !tri->reported &&
  !found_tri->reported`, and the sibling edge index must be **recomputed** for
  `found_tri` (the two halves may be different letters). Checking both `reported`
  flags is what makes the predicate symmetric, so the emitted tile is independent
  of BFS visit order. The resulting 4-corner winding is observable through dot
  indices — do not normalise or sort it.
- `inbounds` is fully conjunctive over all three vertices, so a straddling
  triangle is rejected whole and never explored through. That raggedness is
  exactly why `gridTrimVigorously` runs afterwards.
- Hat reflection: reflected hats are exactly "hat #3 of an H metatile", handled
  by swapping `kite.left`/`kite.right` before the boundary walk and reversing the
  vertex list after — so every face reaches `TilingBuilder.face()` clockwise, as
  it requires.

## Findings during implementation

### An upstream crash, found by building the fixture matrix

Generating the Part-A fixtures aborted the **C harness itself**:

```
Assertion failed: (0 < size && size <= DSF_MAX && "Bad dsf size"),
function dsf_new_internal, file dsf.c, line 61
```

`grid_new_desc(GRID_PENROSE_P2, 3, 3, rs)` happily returns a desc — `"50A"` —
and `grid_new` on that same desc then dies, because the patch has no landlocked
dots at all and `grid_trim_vigorously` calls `dsf_new(0)`.

Observed boundary over seeds `"0".."3"`:

| variant | aborts | clean from |
|---|---|---|
| P2 | 3×3 on seeds 0, 2 (fine on 1); **every** height at width 3 | 4×4 |
| P3 | 3×3 on seed 1; 4×4 on seeds 1, 3 | 5×5 |

The aborting cases have a tell: their descs are exactly the ones with a
**single** coordinate letter — `"50A"`, `"01Y"`, `"71X"`. One letter means the
prototype was never extended, which means the BFS never ran, which means
`penrose_initial`'s seed triangle landed outside the bounding box
(`penrose.c:710-716` adds it to `placed` but only *enqueues* it if `inbounds`).
An empty queue emits nothing, so the grid comes out empty.

Three things make this worth recording rather than just routing around:

1. **It is seed-dependent, not size-dependent.** The same `(type, w, h)`
   succeeds or aborts depending on the draw, so no amount of parameter
   validation can catch it by inspection.
2. **It is reachable from the game.** `loopy.c:713-717` rejects only
   `w < amin || h < amin` and `w < omin && h < omin`, and `GRIDLIST` declares
   both Penrose variants with `amin = omin = 3`. Custom → Penrose → 3×3 passes
   validation and reaches `grid_new`.
3. **D6b anticipated it before it was found.** That decision — throw rather than
   hand back an empty grid — was taken because "an empty grid surfaces far from
   its cause". This is precisely that failure, one layer further down. The TS
   port raises `GridTrimmedAwayError` where upstream aborts, which is strictly
   better behaviour and needs no special-casing.

**Handoff to `add-loopy-ts-port`:** Loopy must decide what to *do* here — raise
the effective minimum for the Penrose types, or catch the error and regenerate
with a fresh seed (which is the more faithful fix, since the failure is per-seed
rather than per-size). Left out of this change deliberately: it is a game-level
policy decision, and the geometry layer's job is only to fail honestly.

## Risks

- **Table transcription volume** (~3,500 lines). Mitigated by D3 (scripted, not
  typed) and D1 part A (a wrong entry fails the differential loudly).
- **RNG stream desynchronisation.** The highest-consequence risk, because it is
  silent — a desynchronised stream produces a valid tiling. Mitigated by D2 as a
  stated rule, D1 part B as the check, and D4's targeted test for the one path no
  ordinary fixture reaches.
- **Penrose's hand-transcribed maps** (D3.3) are the one place a human types 96
  entries. Mitigated by exhaustive `Letter`-union typing and D1 part A.
- **Periodic regression.** The 14 periodic tilings and Pearl's square path must
  stay byte-identical; change 1's differential is the bar and must stay green.
- **No acceptance surface.** Like change 1, nothing renders until Loopy. This is
  stated in the proposal rather than hidden, and it is why the differential is
  index-exact rather than property-based.
