# Loopy port ‚Äî where this leaves off

`extend-grid-tilings` is change **1 of 3** delivering Loopy, the last unported
game. This note is the handoff for the next session; the strategic context is in
[`AGENTS.md`](../../../../AGENTS.md) and the findings are in this change's
[`design.md`](./design.md) (see especially the **Addendum**).

## Why Loopy is the whole endgame

Loopy is the only remaining `puzzle(...)` in `puzzles/CMakeLists.txt` without
`TS_PORTED`. Every other `.c` still in the tree is either Loopy's dependency
subtree or already orphaned:

- `grid.c`, `loopgen.c` ‚Üí consumed only by `loopy.c`
- `penrose.c`, `penrose-legacy.c`, `hat.c`, `spectre.c` ‚Üí only by `grid.c`
- `tree234.c` ‚Üí only by that subtree plus the already-orphaned `latin.c`

So finishing Loopy retires the C engine entirely. Total remaining scope is
~14,300 lines of C, of which **~3,500 are generated lookup tables**
(`hat-tables.h` 2120, `spectre-tables-auto.h` 1220, `-manual.h` 160) that should
be transcribed by script, not by hand.

## Change 2 ‚Äî `add-aperiodic-tilings` (next)

Everything RNG-bearing and desc-round-tripping, which is exactly what change 1
deliberately excluded (the split is at the RNG boundary ‚Äî design D1).

**Suggested order ‚Äî easiest logic first, to establish the table-transcription
approach before the hard recursion:**

1. **`hat.ts`** (~891 lines logic + 2,391 of tables). *Simplest control flow*: a
   14-state serpentine kite enumerator, no BFS, no `tree234`. Two hazards:
   `try_step_coords_metamap`'s unbounded `for (depth = 2;;depth++)` at
   `hat.c:691` needs this repo's [`engine/retry-limit.ts`](../../../../src/native/engine/retry-limit.ts)
   treatment, and `hat.c:834`'s `(v.x*2 + v.y)/3` is C truncating division
   (`Math.trunc`).
2. **`spectre.ts`** (~599 + 1,380 table lines). Tables are generated by X-macro
   expansion (`HEX_LETTERS(Z)`), so they must be **flattened by script**.
   Mutual recursion `spectrectx_step` ‚Üî `spectrectx_step_hex`. The 3-colouring
   sidecar (`spectre.c:381-403`) is dev-harness only ‚Äî droppable.
3. **`penrose.ts`** (~894 lines, no external table file ‚Äî the "tables" are two
   big nested switches transcribable as object maps). *Logically hardest*:
   genuine recursion, BFS with a visited set, and the subtle half-tile pairing
   at `penrose.c:736-755` that emits each rhomb exactly once when both triangle
   halves are present.

Then the `grid.c` side: the four `grid_new_*` wrappers and their callbacks,
`gridNewDesc`/`gridValidateDesc`, and **`gridTrimVigorously`** ‚Äî which upstream
implements with an `O(numDots¬≤)` *dense* matrix (millions of entries for a 10√ó10
spectre patch); implement it as a `Map` keyed on the ordered dot-index pair
(design D8).

Also needed: **`n_times_root_k`** (`misc.c:569`) ‚Äî an exact bitwise
`round(n¬∑‚àök)`, written that way specifically to avoid FP rounding. Do **not**
substitute `Math.round(n * Math.sqrt(k))`.

### Carry these forward from change 1

- **Negative zero is the live hazard.** It nearly shipped in floret and is
  invisible to every check except a structural one (see the Addendum). The
  aperiodic tilings use signed basis vectors throughout ‚Äî expect it.
- **Reuse the differential pattern.** `puzzles/auxiliary/grid-trace.c` already
  has `--all` and `--incentres` modes and is retained for this purpose. Extend
  the fixture matrix rather than inventing a new harness.
- **`penrose-legacy.c` is deliberately dropped** ‚Äî reached only via
  `if (*desc == 'G')` (the pre-rewrite desc format), which `grid_new_desc_penrose`
  never emits. It is also the only float-bearing tiling code. Reject `'G'` descs.
- **Replay touches the RNG**: penrose and spectre fall back to
  `random_new("dummy", 5)` when a stored coord string runs out
  (`penrose.c:583-591`, `spectre.c:309-330`). Divergence there is silent ‚Äî same
  desc, different tiling.
- **The parallel-agent pattern worked.** Give each agent an exclusively-owned
  file; keep the only shared edit a dispatch switch, where contention is a
  compile error rather than a silent wrong answer.

## Change 3 ‚Äî `add-loopy-ts-port`

Facts established by the `loopy.c` survey, worth not re-deriving:

- **No backtracking at any difficulty.** `solve_game_rec` is misnamed ‚Äî four
  deduction rungs run to a fixpoint. `DIFF_TRICKY` isn't its own solver; it only
  unlocks extra branches inside `dline_deductions`. This fits the shared
  [`runDeductionFixpoint`](../../../../src/native/engine/deduction-fixpoint.ts)
  runner, and makes Loopy a strong Palisade-bar candidate for a later
  `add-loopy-hint`.
- **Three byte-parity traps, two of them upstream bugs a clean port would
  silently "fix"** ‚Äî changing which boards generate:
  1. `face_setall_identical` sets `retval = false` and never reassigns it, so it
     always reports "no progress" even when it changed the board.
  2. `parity_deductions` receives `(clue - yes) % 2`, which is **negative** when
     `clue < yes`. C's truncating `%` yields `-1`, which is truthy, so the XOR
     always gives `LINE_YES`. TS `%` truncates identically, so a *literal* port
     matches ‚Äî but the hygiene fix `((x % 2) + 2) % 2` breaks it.
  3. Two deliberate `switch` fallthroughs in `interpret_move` (non-stylus
     YES/NO fall into the `'u'` case).
- **No keyboard input and no drag** ‚Äî mouse clicks only, so `gridNearestEdge` is
  the entire input path. There is an `autofollow` preference (off/fixed/adaptive).
- `loopy.c` includes `tree234.h` but never uses it ‚Äî don't port a dependency for it.
- `MAX_FACE_SIZE 14` is a hard assert in the solver.
- **Loopy has its own grid enum**, ordered differently from `GRIDGEN_LIST` and
  frozen into saved game IDs. Keep both orderings and the mapping between them
  (design D9). Per-type *minimum* sizes live there too (`amin`/`omin`), not in
  the geometry.

Stage 2 of this change deletes `puzzles/loopy.c` and its now-unused subtree.

## Change 4 ‚Äî `retire-c-engine` (follow-up)

Owner-chosen as a **separate** change rather than folded into Loopy's stage 2:
remove the already-orphaned leaves (`latin.c`, `matching.c`, `combi.c`, `dsf.c`,
`findloop.c`, `sort.c`, `laydomino.c`, `divvy.c`), `webapp.cpp`, the Emscripten
build, the worker's WASM path, and the `USE_TS_LEAVES` machinery. That is an
architectural change deserving its own `design.md`.

Note `divvy.c` has its own blocker ‚Äî the unfinished, non-catalogued `separate`
puzzle is its last consumer.

## Acceptance status of change 1

Accepted 2026-07-18 **on the explicit basis that there was nothing to
acceptance-test**: no game consumes these tilings yet, so there is no rendered
surface to drive. Its assurance comes from the index-exact C differential (309
assertions over 44 fixtures) and the gate ‚Äî *not* from a human driving it.
**The first real acceptance test of this code is Loopy itself.** If something is
subtly wrong in a tiling, that is where it will show up.
