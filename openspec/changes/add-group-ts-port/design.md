# Design — add-group-ts-port

## Context

Group (`puzzles/unfinished/group.c`, ~2530 lines) is a Latin-square puzzle whose
extra constraint is *group-ness*: the filled grid must be a valid Cayley table,
i.e. Latin **and** associative (from which identity and inverses follow — the
proof is in `check_errors`). Upstream left it in `unfinished/` purely on taste
("too esoteric… not to mention _hard_"), not because it is incomplete: it has a
full solver, generator, renderer, save format and two genuine gameplay aids.

The decisive fact for cost is that **`src/native/engine/latin.ts` already exists**
and is the shared engine for Towers, Solo, Unequal, Keen and Undead. Group's
`solver()` calls `latin_solver_main` exactly as those do; the port is "supply two
user-solvers and a validator to `latinSolver`", not "write a solver". Most of the
remaining work is faithful data transcription (the group table) and the two
Group-specific UI aids.

Two surveys of `group.c` (solver/generator; UI/input/render) inform the decisions
below, recorded here rather than rediscovered mid-port.

## Decisions

### D1 — Reuse `latin.ts` wholesale; Group is two user-solvers + a validator

`group.c`'s solver is:

```c
latin_solver_main(&solver, maxdiff,
    DIFF_TRIVIAL, DIFF_HARD, DIFF_EXTREME, DIFF_EXTREME, DIFF_UNREASONABLE,
    group_solvers, group_valid, NULL, NULL, NULL);
```

Mapped onto `LatinSolverConfig` (`latin.ts`), with Group's difficulty enum
`{ TRIVIAL=0, NORMAL=1, HARD=2, EXTREME=3, UNREASONABLE=4 }`:

- `diffSimple = TRIVIAL(0)`, `diffSet0 = HARD(2)`, `diffSet1 = EXTREME(3)`,
  `diffForcing = EXTREME(3)`, `diffRecursive = UNREASONABLE(4)`.
- `usersolvers = [null, solverNormal, solverHard, null, null]` — Group's own
  deductions fire only at Normal and Hard; Extreme is generic set-elimination +
  forcing, Unreasonable is the shared guess-and-verify recursion.
- `valid = groupValid` — associativity of the *completed* grid.

`latin.ts`'s `applyRung` already runs multiple built-ins at one rung (Extreme
runs both the extreme set-elimination and forcing, matching `diffSet1 ==
diffForcing`), and already threads the game's user-solver *before* the built-in
at each level. Group needs **no** change to `latin.ts`.

Everything the two user-solvers touch is on the existing `LatinSolver` surface:
`o`, `grid`, `place(x,y,n)`, the public `cube` buffer with `cubepos`/`cubeGet`.
`solver_hard` mutates candidates directly (`cube(i,j,j+1) = false`) → `solver.cube[
solver.cubepos(i, j, j+1)] = 0`. `find_identity` reads `grid`/`o`. Keep upstream's
`(x·o+y)·o+(n−1)` cube indexing verbatim (playbook §2.2 — re-deriving transposes
diverges the differential), which is exactly what `cubepos` already is.

### D2 — Transcribe the group data table verbatim

`groupdata[]` (89 groups, orders 2–26) and the `groups[]` index-by-order table are
generated by upstream's `group.gap` (GAP computer-algebra output). They are
**data, not logic** — port them as an `as const` TS table with the same fields
(`autosize`, `order`, `ngens`, `gens`) and the same order. `autosize` is unused by
the C game logic (informational) but carry it for provenance. The generator's BFS
decompresses `gens` (a string of `w`-char rows, one per generator) into the full
canonical table; that BFS ports directly.

Risk: a transcription slip in a 26×… data blob is silent until the differential
catches a wrong board. Mitigation: the byte-match differential (D8) runs every
order the fixtures cover, and the table can be checked once against the C by
dumping `groupdata` from `group-trace.c`.

### D3 — Identity mode and the element-numbering remap

`params.id` ("show identity") toggles two coupled behaviours:

- **Element numbering / char mapping.** With identity shown, element 1 (the
  identity) displays as `e` and the rest as `a,b,c,d,f,g,…` (identity pulled to
  the front); with it hidden, elements are `a,b,c,d,e,…` in order. This is the
  `E_TO_FRONT`/`E_FROM_FRONT` swap inside `FROMCHAR`/`TOCHAR`. Model it as a pair
  of pure helpers `toChar(n, id)` / `fromChar(c, id)` — it affects the desc's
  *solution* aux string, keyboard input, `game_request_keys`, and rendering, but
  **not** the grid desc (which is decimal, D10).
- **Generation.** In identity mode the shuffle fixes element 1 in place
  (`shuffle(soln2+1, w-1)`); otherwise all `w` elements shuffle, and the
  generator additionally blanks the whole identity row/column **plus one more
  random row/column** so the player can't read off the identity for free — with a
  solver re-check that the puzzle still grades at `diff`.

`validate_params` forbids two identity-hidden combinations that can't be made
(both because such puzzles would be Trivial-only, and a Trivial puzzle can't hide
its identity): **`!id && diff == TRIVIAL`** and **`!id && w == 3`**. Port both
guards with their reasons.

### D4 — Which solver techniques ship, and which TODO ones are deferred

Ships (all five difficulties): Trivial (generic Latin), Normal (Group
associativity forward-deduction + identity row/col fill), Hard (identity-hidden
candidate elimination), Extreme (generic extreme-set + forcing), Unreasonable
(recursion). This is upstream's full shipped solver.

Deferred, explicitly out of scope (the header's own TODO list, never implemented
upstream): inverse-based deductions (`gh=e ⇒ hg=e ⇒ …`), hard-mode associativity
over *possible* rather than *definite* entries, and element-order inference. They
are a future strengthening change, not a port gap — the difficulty curve upstream
shipped is the one without them (playbook rule 3: a *weaker* solver is the shipped
curve, not a defect).

### D5 — Port both visual aids (row/col reorder + subgroup dividers)

Two Group-specific aids that are genuine gameplay value, not chrome:

- **`sequence`** — the display order of the `w` elements. Dragging a row/column
  *header* (the legend) repositions an entire element's row and column, letting a
  player group a subgroup and its cosets together. `interpret_move` emits a
  `D<num>,<pos>` move; `execute_move` rebuilds `sequence`. The `hx/hy` highlight
  and diagonal multifill selection track through `sequence`, which `game_ui`
  bookkeeping (`ohx/ohy/odx/ody/odn`) manages.
- **`dividers`** — thick lines a player drops between adjacent rows/columns to
  mark subgroup boundaries; a `V<i>,<j>` move, auto-cleared when the two elements
  it separates are dragged apart.

These carry real design weight (they are why the puzzle is *about* group
structure, not just a Latin square) so they are in scope. They complicate input
(header-drag detection, cursor mapping through `sequence`) — budget for that. The
diagonal **multifill** (drag from a highlighted cell along a diagonal to fill a
whole `a·a`-style diagonal at once) is part of the same aid family and ships too.

### D6 — Move model: a discriminated union of ops

Per the repo's established pattern (D5 in the Loopy port; Pearl/Tracks/Map/Rect),
model moves as a discriminated union that `interpretMove` builds and `executeMove`
consumes exhaustively, not upstream's parsed string. Ops:

- `{ kind: "set"; cells: {x,y}[]; n }` — fill/clear (multifill carries several
  cells); `n = 0` clears.
- `{ kind: "pencil"; cells: {x,y}[]; n }` — toggle a pencil mark.
- `{ kind: "solve"; grid }` — the full solution.
- `{ kind: "reorder"; num; pos }` — a `sequence` drag (upstream `D`).
- `{ kind: "divider"; i; j }` — toggle a divider (upstream `V`).

Upstream's `'M'` "fill every pencil mark everywhere" is a standalone-solver
diagnostic aid, not a play move; **drop it** (record the omission). The save
format is ours, so the wire spelling is free to be the union.

### D7 — `findMistakes` via re-solve; Group ships Check & Save

Group is uniquely solvable, so the playbook (§3.5) requires `findMistakes`.
`check_errors` already computes a live internal-consistency overlay (Latin
duplicates + associativity-failure annotations) for rendering — but that answers
"is the board self-contradictory?", not "does it contradict the unique solution?".
For Check & Save, implement `findMistakes` the Galaxies/Keen way: re-solve from the
immutable givens and flag any user entry that disagrees with the unique solution.
Keep `check_errors`'s associativity/Latin overlay as the *render-time* error
display (it is what draws the red `(ab)c`/`a(bc)` annotations and duplicate
highlights); the two are complementary and both ship.

### D8 — Byte-parity policy and the differential

Default to byte-agreement with the C on the generator/solver/codec path, because
it is the **verification mechanism** (the desc depends on the solver's verdict on
every intermediate board, so one differential assertion validates generator,
solver and codec together), not because fidelity is a goal in itself — the repo's
standing policy. A new `puzzles/auxiliary/group-trace.c` dumps the generated desc
for `(w, diff, id, seed)` tuples; `group-differential.test.ts` asserts the TS
`newDesc` reproduces each desc byte-for-byte **and** that the TS solver grades each
C board at exactly the recorded difficulty.

Matrix: cover several sizes (small like `4x`, mid like `6x`/`8x`, and at least one
large like `12x`), **both** identity modes, and every difficulty that size admits
(mind the difficulty-downgrade exceptions in `new_game_desc` — a fixture asking for
Extreme at `w=4` actually generates Normal). Display code (rendering, dividers,
multifill visuals) is out of byte-parity scope per repo policy — it targets neat
visuals, tested at tier 2.5.

### D9 — The catalog question is the one genuine open decision (owner)

Group was **never** in upstream's shipped catalog. Bringing it in is a deliberate
divergence, and there is a clean precedent: `separate` was likewise an
`unfinished/` puzzle and is now a full catalog citizen (`TS_PORTED`, no `.c`).

**Recommendation: full catalog citizen, same as every other game.** This fork's
stated goal is to grow *beyond* upstream's set; Group is complete, its icons
already exist, and `separate` set the pattern. Upstream's reason for shelving it
(too hard/esoteric for a general audience) is a *presentation* concern, and this
fork's audience self-selects into a puzzle collection.

But this is genuinely the owner's call — it is the only decision here with a real
product trade-off (surfacing a puzzle most players will bounce off vs. hiding a
finished game behind nothing). Alternatives if "full citizen" is rejected: ship it
registered-but-unlisted (reachable by URL only), or defer the catalog flip while
still landing the port. **Flagged, not presumed** — see Open Questions.

### D10 — Desc codec: decimal clues, not single-char

Unlike most Latin games, Group's element values run 1–26, so the **grid desc uses
decimal numbers** (`sprintf("%d")`) with `a`–`z` for runs of 1–26 blanks and `_`
to separate a number from adjacent content where a digit boundary would otherwise
be ambiguous (`encode_grid`/`spec_to_grid`/`validate_grid_desc`). The display
letters (`TOCHAR`) never appear in the grid desc — only in the *solution* aux
string and on screen. Port the run-length encoder verbatim, including the "no
leading/trailing `_` around a number at the very corners" nicety and the
`run > 26` split.

## Risks

- **Data-table transcription.** 89 groups of generator strings; a slip is silent
  until the differential fires. Mitigated by dumping `groupdata` from
  `group-trace.c` and by breadth in the fixture matrix (D2, D8).
- **The two visual aids are the most input-heavy part.** Header-drag detection,
  cursor/highlight mapping through `sequence`, divider auto-clear on drag, and
  diagonal multifill are where the fiddly bugs live — not in the solver. Test
  input transitions at tier 1 and the aids' rendering at tier 2.5.
- **Esotericness is a product risk, not a technical one.** Even done perfectly,
  most players won't engage. That's D9's decision, surfaced honestly.
- **No hint ships here**, so Group's first release is less capable than Palisade.
  Established sequencing, stated out loud.

## Open Questions

- **Catalog surfacing (owner).** Full catalog citizen (recommended, per the
  `separate` precedent), registered-but-unlisted, or land the port with the
  catalog flip deferred? The only decision here with a real product trade-off.
- **Keep or drop the `'M'` diagnostic move?** Recommendation: drop (D6) — it is a
  standalone-solver aid, not gameplay. Trivial to restore if wanted.
