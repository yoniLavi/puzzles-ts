## Context

Layer-3 test discipline has been on the AGENTS.md docket since the project started but has never been built. With seam #2 (`combi.c`) just landed and the AGENTS.md "C is never deleted until rewrite complete" policy formalised, the soak is now load-bearing: it's the only end-to-end gate that proves the hybrid build matches the pure-WASM build, and the policy explicitly defines "complete" in terms of soak runs.

This is the **biggest single piece of new infrastructure** since the random.c port. It touches the C side (deterministic-solve-trace surface), the TS side (runner, corpus, comparison), the build-pipeline (running both modes from one script), and the developer workflow (pre-commit + pre-push hooks). The proposal is conservative on per-module specifics so we can iterate during implementation; the durable design decisions are below.

## Goals / Non-Goals

**Goals**
- End-to-end fidelity gate: hybrid (`USE_TS_LEAVES=ON`) and pure-WASM builds produce byte-identical canonical outputs for a fixed `(puzzle, preset, seed)` corpus.
- Smoke subset runs in pre-commit (per the user directive) with a strict latency budget.
- Full soak runs in pre-push.
- Corpus is versioned in-repo, regenerated by running the pure-WASM mode and snapshotting.

**Non-Goals**
- Performance benchmarking. Correctness only.
- Per-puzzle test ports beyond what's already in `puzzles/auxiliary/`.
- A pretty UI. Terminal output + exit code is the contract.
- Covering puzzles whose solver isn't exposed through midend (those puzzles get a "generates-without-crashing" check, not a solve-trace check; see Decisions).

## Decisions

### Decision: Canonical output is `{ gameId, postGenerateStateHash, solvedStateHash }`

Per `(puzzle, preset, seed)` triple, the runner records three SHA-256 hex digests:

1. **`gameId`**: the string returned by the engine after `new_game_id_from_random_seed`. RNG drift surfaces here.
2. **`postGenerateStateHash`**: a SHA-256 of the engine's serialised game state right after generation (the same byte-string that powers `save game`). Catches drift in any allocation-order-dependent generator code.
3. **`solvedStateHash`**: a SHA-256 of the engine's serialised game state *after* `midend_solve`. Catches solver drift via its observable output. Falls back to `"unsolvable:<err>"` if `midend_solve` returns an error; falls back to `"timeout"` if the solve exceeds the wall-clock budget. Driven from `webapp.cpp` using the existing midend API; no edits to `midend.c` (see AGENTS.md "Upstream policy").

**Why three hashes, not one**: different layers of drift surface at different points. Bundling them obscures *what* drifted; separating them makes failures diagnosable from the test output alone.

**Why hashes, not raw state**: keeps the committed corpus tiny (3 hex digests per row × hundreds of rows ≈ <100 KB), readable in diffs, and trivial to compare. When a hash drifts, the second-pass debug step is "rerun with `--dump-state`" to get the raw bytes.

**Alternatives considered**:
- Game ID only. Catches RNG drift but not solver drift. Insufficient.
- Full state diff in-corpus. Bloats the repo; harder to read diffs.
- A single combined hash. Loses diagnosability.

### Decision: Solve hash via `webapp.cpp` calling existing midend API

Per AGENTS.md "Upstream policy", `puzzles/midend.c` is immutable; we don't add anything to it. Instead, expose a new Embind function in `puzzles/webapp.cpp` (our adapter, freely editable) that drives the existing `midend_solve` and hashes the resulting solved state's serialised bytes:

```cpp
EMSCRIPTEN_BINDINGS(soak_extras) {
    function("solveAndHashState", &solve_and_hash_state);
}

std::string solve_and_hash_state(intptr_t midend_handle) {
    midend *me = reinterpret_cast<midend *>(midend_handle);
    const char *err = midend_solve(me);
    if (err != nullptr) return "unsolvable:" + std::string(err);
    char *serialised = midend_serialise_to_string(me);
    std::string hash = sha256_hex(serialised);
    sfree(serialised);
    return hash;
}
```

(Pseudocode — exact serialisation helper TBD during implementation; midend exposes save/load primitives we can drive without modifying them.)

**Why this shape**:
- One soak-specific Embind addition in our adapter; no edits to upstream C.
- Works uniformly across all ~40 puzzles via the existing midend interface.
- The hash captures *what was solved to*, which is the right fidelity bar — two builds that solve to different states have drifted, regardless of whether their internal solve traces differ.

**Trade-off vs the earlier "hash the move trace" idea**: hashing the solved state is a slightly coarser signal than hashing the move sequence (two algorithms that arrive at the same state via different move sequences will hash identically). That's acceptable because the soak's job is to catch *observable* drift; an internal move-sequence change that arrives at the same answer isn't drift we care about. The earlier formulation would have required modifying midend.c to expose move-sequence hooks; under the upstream policy, that's not on the table.

**Risk**: some puzzles' solvers are heuristic and may take an unbounded amount of time on certain seeds. Mitigation: hard timeout via the existing midend timer mechanism if available, or wall-clock check from the Embind side; on timeout, the hash records `"timeout"`. Documented as a known coverage gap for those (puzzle, seed) cases.

### Decision: Smoke subset is `(2 puzzles) × (1 preset each) × (3 seeds) × (2 modes) = 12 runs`

Pre-commit smoke runs:
- **Puzzles**: one with an RNG-heavy generator and a reachable solver (e.g. **Solo** or **Loopy**) plus one with a simple combinatorial structure (e.g. **Flip** or **Cube**).
- **Presets**: the default `Easy` (or smallest non-trivial) preset for each.
- **Seeds**: three fixed values, including the all-zeros seed and one previously-observed canary seed (the random spec already records `randomSeed=3x3#786954740169111` for Solo — reuse).
- **Modes**: pure-WASM + hybrid.

Latency budget: **≤ 8 seconds total**, including WASM module instantiation overhead. If we blow this budget, reduce the corpus (drop a seed) rather than skipping pre-commit entirely.

**Why so small**: pre-commit running on every save-and-commit. If the gate is even 30 seconds, developers reach for `--no-verify` and the gate becomes ceremonial. Eight seconds is annoying-but-tolerable; reflects the gate's "smoke" purpose (not full coverage).

### Decision: Full soak is `(every puzzle) × (every preset) × (5 seeds) × (2 modes)`

Pre-push runs:
- **Puzzles**: every puzzle in `gamelist[]`.
- **Presets**: every preset exposed by each puzzle's `game.encode_params` table.
- **Seeds**: 5 fixed values per `(puzzle, preset)`. The smoke triple plus two more, locked at corpus-recording time.
- **Modes**: both.

Estimated runtime: ~40 puzzles × ~5 presets × 5 seeds × 2 modes × ~100ms median per generate+solve ≈ 200 seconds. Realistic; below the threshold where push gets abandoned.

**Why both halves matter**: smoke catches *fast* regressions (per-commit, low-coverage); full catches *thorough* regressions (per-push, high-coverage). The full soak is the gate that lets us eventually declare "C can be deleted" per the AGENTS.md trigger.

### Decision: Corpus is recorded against pure-WASM; hybrid is asserted equal

The committed `corpus.json` records canonical outputs from a pure-WASM build. The soak then runs the corpus under hybrid and asserts every output is identical. This makes C the source of truth — exactly the symmetry we want (C is the un-displaced impl; TS bridges have to match it).

A separate developer command (`npm run soak:rerecord`) regenerates `corpus.json` from a fresh pure-WASM build. Used when a) the corpus's puzzles/presets/seeds change deliberately, or b) an upstream subtree merge legitimately shifts pure-WASM output (e.g., a generator improvement). Re-recording is loud and intentional, not automatic.

**Alternatives considered**:
- Re-record on every soak run. Defeats the purpose — never catches drift.
- Compare hybrid against hybrid (no pure-WASM in the loop). Loses C as the fidelity anchor; useless for the "delete C" trigger.

### Decision: WASM artifacts cached aggressively across modes

The soak driver expects `/build/wasm-pure/` and `/build/wasm-hybrid/` as separate artifact directories — one cmake out-of-source build per mode. Incremental rebuilds within a mode are ~2s (per the build-pipeline spec); a cold rebuild is ~2 min and only happens when source changes invalidate the cache.

`scripts/build-emcc.sh` gains a `--out-dir` flag (or env var) so the soak can target distinct directories without colliding with `npm run build:wasm`'s default `/build/wasm/`.

**Why two directories, not flag-toggled rebuilds**: keeps the smoke subset fast (no rebuild per pre-commit), and makes "the current state of both modes" inspectable on disk. A flag-toggled build would force a rebuild whenever the wrong mode was last built — failing the smoke gate's latency budget on every other commit.

## Risks / Trade-offs

- **Pre-commit budget creep.** The smoke step is 8s today; as we add seams, the per-mode WASM initialisation cost might grow. Mitigation: explicit budget assertion in the runner — emit a warning to stderr if smoke exceeds 12s, and a hard failure at 20s with a "scope down the corpus" message.
- **Solver non-termination on hostile seeds.** A puzzle's heuristic solver might loop forever on a pathological seed. Mitigation: hard timeout, documented. The corpus avoids pathological seeds by construction (recorded against a stable build that terminated).
- **Embind boilerplate.** Exposing `solveAndHashState` requires careful Embind work, including pointer-handle management. This is the same machinery as `wire-random-to-wasm`'s `Module.tsRandomBridge`. Mitigation: reuse the pattern, don't reinvent.
- **The corpus becomes load-bearing.** If `corpus.json` is silently wrong (e.g. recorded against a temporarily-broken pure-WASM), every soak run will green-light a regression. Mitigation: re-recording is explicit; the re-record command prints a diff against the previous corpus before overwriting, and the resulting diff is reviewed in the PR.
- **Coverage gap for puzzles with no reachable solver.** `midend_solve` returns an error for those; `solvedStateHash` records `"unsolvable:<err>"`. The soak still catches RNG and post-generate drift for those puzzles, just not solver drift. Acceptable until a future per-puzzle port exposes a solver path.

## Migration Plan

Incremental:

1. Build the soak runner against the existing pure-WASM build alone. Record the first corpus. Soak runs `npm run soak:smoke` against pure-WASM-only, asserts canonical outputs match the corpus. Half the value of the full soak; gives us per-puzzle determinism right away.
2. Once `add-use-ts-leaves-umbrella-flag` is implemented, extend the soak to run both modes and compare. Hybrid-vs-pure becomes the real fidelity assertion.
3. Wire pre-commit (smoke) and pre-push (full).

Rollback at any stage: the soak is additive; nothing downstream depends on it.

## Open Questions

- **Which two puzzles get the smoke slot?** Solo + Flip is my lean (one heavy generator, one trivial one). Confirm during implementation by measuring actual runtimes.
- **Should the soak run against the dev-server hot-reload build, or only against production-style `npm run build:wasm` artifacts?** Dev-server is closer to "what the developer is actually testing," but production artifacts are what ships. Lean production; confirm.
- **Pre-push hook strategy.** Husky supports pre-push. Alternative: a `gh workflow` that runs the full soak on PR open. Lean pre-push (local fast-feedback) initially, add CI later.
- **Game ID stability under engine refactor.** If an upstream subtree merge legitimately changes game-ID format (rare but possible), the corpus needs re-recording. Document the re-record-and-review workflow.
