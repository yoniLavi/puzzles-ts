# retire-c-engine

## Why

Loopy was the last game (`add-loopy-ts-port`). Every catalogued game now runs on
the native TypeScript engine; no game is served by C/WASM. The C engine, the
Emscripten build that produced it, the worker's WASM dispatch path, and the
`USE_TS_LEAVES` leaf-bridge machinery are therefore **dead weight** ŌĆö still
compiled, still maintained, consumed by nothing at runtime.

This change removes them. It was deliberately kept separate from the per-game
ports (each port's proposal says so): it is an architectural teardown, not a
game port, and it deserves its own design because the risk is entirely in the
build wiring ŌĆö the catalog and the in-app manual are *currently* generated by
the same Emscripten build that compiles the (now unneeded) wasm, so that
generation has to survive the teardown.

## What Changes

- **Delete the runtime-orphaned C sources.** After Loopy the C files with no
  remaining consumer are the shared leaves and the C midend/drawing stack:
  `tree234.c` + `latin.c` (latin uses tree234; every latin-family game is
  ported), `combi.c`, `dsf.c`, `findloop.c`, `sort.c`, `midend.c`, `drawing.c`,
  `misc.c`, `malloc.c`, `tdq.c`, `ps.c`, `draw-poly.c`, `version.c`, the random
  stack, and the null frontends. Confirm each against the build before deleting;
  `matching.c` is both a core source and an auxiliary program.
- **Delete `webapp.cpp`** ŌĆö the Embind adapter between the C games and the TS
  worker. With no C game it adapts nothing.
- **Remove the Emscripten build** ŌĆö `scripts/build-emcc.sh`, `webapp.cmake`, the
  `build:wasm` path, and the `src/assets/puzzles/*.wasm` outputs. **Preserve the
  catalog + manual generation** that currently rides on it (see design D2).
- **Remove the worker's WASM path** ŌĆö `WorkerPuzzle`, the C branch of the
  `createTsEngine` dispatch seam, `assertWasmBridgesCoherent`, and the
  `USE_TS_LEAVES` / `VITE_USE_TS_LEAVES` umbrella plus every per-module
  `USE_TS_*` flag. These exist only to gate C-internal leaf bridges that no
  longer have a C side.
- **Re-evaluate whether the Web Worker survives at all** (flagged in the
  `ts-migration` spec). It exists for heavy WASM; pure-TS games may not need a
  worker thread. Decide here (design D3).

Explicitly **not** in this change:

- **`divvy.c` stays only if still coupled** ŌĆö it was already deleted when
  Separate shipped (2026-07-04), so verify it is truly gone before assuming the
  historical "separate blocker" applies; it does not.
- **`puzzles/LICENCE`** stays permanently (MIT obligation).
- **The unfinished puzzles** (`puzzles/unfinished/`) and their build are out of
  scope ŌĆö they are governed by the per-game `add-{group,slide,sokoban,numgame,
  path}-ts-port` changes.

## Impact

- Affected specs: `ts-migration` (the migration reaches its terminal state ŌĆö C
  fully gone), `ts-engine` (only one engine surface remains; no dispatch
  branch), `build-pipeline` (the Emscripten/wasm build and the leaf-flag
  machinery are removed, replaced by a wasm-free asset build).
- Affected code: large but mechanical deletions across `puzzles/`,
  `src/puzzle/worker.ts`, `src/puzzle/` (the `WorkerPuzzle` path), `scripts/`,
  `vite-plugins/`, and the CMake tree; plus whatever now generates `catalog.json`
  and the manual.
- Risk is in the **build wiring**, not logic ŌĆö do it with the app running so a
  broken asset path surfaces at once. Reversible via git if a catalog/manual
  dependency was missed.
